# TGU_Robocore_2027 项目结构与代码风格规范

> 适用仓库：`TGURM304/TGU_Robocore_2027`  
> 文档目的：统一项目目录规划、模块职责、命名方式、C++ 代码风格和后续扩展方式，方便后续开发自瞄、导航、串口通信、相机、Foxglove 可视化等模块。

---

## 1. 项目定位

`TGU_Robocore_2027` 是一个面向 RM 机器人视觉/导航的 C++ 框架。当前仓库主要包含：

- `io`：硬件通信和设备输入输出，例如串口、工业相机接口。
- `tools`：通用工具库，例如 CRC、Logger、toml++、Foxglove 通信封装。
- `task`：可执行入口、测试程序、功能验证程序。
- `app`：上层应用逻辑，例如自瞄、导航、决策等。
- `config`：运行参数配置，例如相机参数、阵营颜色、外设配置。

项目应遵循“底层驱动与上层算法解耦”的原则：

```text
config  ->  task  ->  app
             |       |
             v       v
            io  <-> tools
```

含义：

- `io` 不应依赖 `app`。
- `tools` 应尽量独立，不依赖具体业务。
- `app` 可以调用 `io` 和 `tools`。
- `task` 负责把配置、设备、算法、测试流程组织起来。
- `config` 只存放参数，不写业务逻辑。

## 3. 目录职责规范

### 3.1 `config/`

用于保存可调参数，不写 C++ 逻辑。

推荐放置内容：

- `camera.toml`：相机名称、曝光、增益、内参、畸变参数。
- `game.toml`：敌方颜色、比赛模式等。
- `serial.toml`：串口设备名、波特率、协议开关。
- `foxglove.toml`：host、port、是否启用可视化。

建议：

- 所有运行时可调参数优先放到 TOML。
- C++ 中不要硬编码相机内参、串口名、曝光值。
- 配置读取失败时必须给出日志提示，并提供合理默认值。

---

### 3.2 `tools/`

用于放通用工具，不绑定具体机器人任务。

已有模块：

| 模块                     | 作用                        |
| ------------------------ | --------------------------- |
| `crc.hpp/.cpp`           | CRC8 / CRC16 计算与校验     |
| `logger.hpp/.cpp`        | 统一日志系统                |
| `tomlpp.hpp`             | TOML 配置读取依赖           |
| `foxglove_comm.hpp/.cpp` | Foxglove WebSocket 通信封装 |
| `foxglove/`              | 第三方 Foxglove SDK/库文件  |

开发要求：

- `tools` 内的代码应尽量不依赖 `io`、`app`、`task`。
- 工具类应做到可复用、可单测。
- 第三方库放在独立子目录中，不要和自己写的工具代码混在一起。
- 修改第三方库前必须注明原因；能不改就不改。

---

### 3.3 `io/`

用于硬件输入输出、通信驱动、设备封装。

已有模块：

| 模块        | 作用                                 |
| ----------- | ------------------------------------ |
| `serial/`   | 基于 Boost.Asio 的结构体驱动串口通信 |
| `hikrobot/` | 海康工业相机接口，占位/待实现        |

开发要求：

- `io` 层只负责“把数据稳定拿进来/发出去”。
- `io` 层不直接写自瞄算法、导航算法、决策逻辑。
- 每个设备建议一个独立子目录，例如：
    - `io/serial/`
    - `io/hikrobot/`
    - `io/mindvision/`
    - `io/cboard/`
    - `io/imu/`
- 设备类要提供明确的生命周期：`open/init`、`close/deinit`、`read/write/spin_once`。

---

### 3.4 `app/`

用于上层业务模块。

建议按功能拆分：

```text
app/
├── auto_aim/       # 自瞄识别、装甲板筛选、PnP、预测
├── navigation/     # 导航、路径规划、定位融合
├── tracker/        # 目标跟踪
├── predictor/      # 弹道/运动预测
└── decision/       # 比赛状态机、策略决策
```

开发要求：

- `app` 可以使用 `io` 提供的数据。
- `app` 可以使用 `tools` 中的 Logger、CRC、配置解析等工具。
- `app` 不应直接依赖具体测试入口，例如 `task/test/*.cpp`。
- 算法模块尽量做到输入输出清晰，方便以后替换模型或调试。

---

### 3.5 `task/`

用于放可执行程序、任务入口、测试程序。

已有内容：

- `test_serial.cpp`：串口通信测试。
- `test_logger.cpp`：日志测试。
- `read_toml.cpp`：TOML 配置读取测试。
- `test_camera.cpp`：相机采图测试。
- `test_foxglove.cpp`：Foxglove 测试，占位/注释状态。
- `sentry.cpp`：主任务入口，占位/待实现。

开发要求：

- `task/test/` 中可以写临时测试代码。
- 正式机器人入口建议放在 `task/sentry.cpp` 或 `task/main_*.cpp`。
- 测试程序可以使用 `std::cout`，正式运行代码优先使用 Logger。
- 每个可执行目标都应在 `task/CMakeLists.txt` 中明确声明。

## 4. CMake 组织规范

### 4.1 顶层 CMake

当前顶层项目使用：

- CMake 最低版本：`3.16`
- C++ 标准：`C++20`
- 依赖：Boost、OpenCV、PkgConfig、Aravis
- 子目录：`io`、`task`、`tools`，`app` 当前处于注释状态

推荐顶层职责：

```cmake
cmake_minimum_required(VERSION 3.16)
project(TGU_Robocore_2027)

set(CMAKE_CXX_STANDARD 20)
set(CMAKE_CXX_STANDARD_REQUIRED ON)
set(CMAKE_CXX_EXTENSIONS OFF)

find_package(Boost CONFIG REQUIRED)
find_package(OpenCV REQUIRED)
find_package(PkgConfig REQUIRED)
pkg_check_modules(ARAVIS REQUIRED aravis-0.8)

add_subdirectory(tools)
add_subdirectory(io)
add_subdirectory(app)
add_subdirectory(task)
```

### 4.2 子模块 CMake

每个模块独立维护自己的 `CMakeLists.txt`。

推荐：

```cmake
add_library(io STATIC
    serial/serial.cpp
    hikrobot/hikrobot.cpp
)

target_include_directories(io PUBLIC
    ${PROJECT_SOURCE_DIR}
)

target_link_libraries(io PUBLIC
    tools
    Boost::boost
    boost_system
    pthread
)
```

注意：

- 优先使用 `target_include_directories()`，少用全局 `include_directories()`。
- 优先使用 `target_link_libraries()` 绑定目标依赖。
- 如果顶层写了 `add_subdirectory(tools)`，则 `tools/` 下必须有 `CMakeLists.txt`。
- `app` 如果开始参与编译，应取消顶层 `add_subdirectory(app)` 的注释，并保证 `app/CMakeLists.txt` 有实际 target。

## 5. 命名风格

### 5.1 文件与目录

推荐使用小写蛇形命名：

```text
serial.cpp
serial.hpp
foxglove_comm.cpp
foxglove_comm.hpp
test_logger.cpp
test_serial.cpp
auto_aim.cpp
auto_aim.hpp
```

规则：

- 目录名：小写蛇形，例如 `auto_aim`、`foxglove_comm`。
- 源文件：`.cpp`。
- 头文件：`.hpp`。
- 测试文件：`test_xxx.cpp`。
- 配置文件：`xxx.toml`。

---

### 5.2 命名空间

已有命名空间：

```cpp
namespace io {
}

namespace tools {
}
```

推荐后续使用：

```cpp
namespace app {
namespace auto_aim {
}
}
```

规则：

- `tools`：通用工具。
- `io`：外设和通信。
- `app`：算法和业务。
- 不要在头文件中使用 `using namespace std;`。

---

### 5.3 类型命名

类、结构体、枚举使用大驼峰：

```cpp
class Serial;
class Logger;
class FoxGloveComm;
template <typename T, size_t N>
class RingBuffer;

enum class LogLevel;

struct LoggerConfig;
struct RecvPackage;
struct SendPackage;
```

规则：

- 类：`UpperCamelCase`
- 结构体：`UpperCamelCase`
- 枚举类：`UpperCamelCase`
- 类型别名：`UpperCamelCase` 或清晰表达用途

---

### 5.4 函数命名

为了后续统一，建议新增代码优先使用小写蛇形：

```cpp
bool is_open() const;
void spin_once();
uint16_t get_crc16(const uint8_t* data, uint32_t len);
bool check_crc16(const uint8_t* data, uint32_t len);
```
规则：

- 普通函数：`lower_snake_case`
- 成员函数：`lower_snake_case`
- 布尔函数：用 `is_`、`has_`、`can_`、`check_` 开头

---

### 5.5 变量命名

局部变量和成员变量使用小写蛇形：

```cpp
int baudrate;
size_t rx_len;
std::string file_path;
```

成员变量建议以下划线结尾：

```cpp
LogLevel level_;
bool console_;
bool file_;
std::ofstream ofs_;
std::mutex mutex_;
```

常量使用大写蛇形或明确的 `constexpr`：

```cpp
static constexpr const char* MODULE = "SERIAL";
static constexpr size_t HEAD_SIZE = 2;
constexpr uint16_t CRC16_INIT = 0xffff;
```

## 6. 头文件与源文件规范

### 6.1 头文件基本格式

推荐：

```cpp
#ifndef TGU_ROBOCORE_2027_SERIAL_HPP
#define TGU_ROBOCORE_2027_SERIAL_HPP
#pragma once

#include <cstdint>
#include <functional>
#include <string>
#include <vector>

namespace io {

class Serial {
public:
    Serial();
    ~Serial();

    bool open(const std::string& device, int baudrate);
    void close();
    bool is_open() const;
};

} // namespace io

#endif // TGU_ROBOCORE_2027_SERIAL_HPP
```

规则：

- 头文件使用 include guard；当前项目也使用了 `#pragma once`，可以保留。
- 头文件只放接口、模板、内联短函数。
- 普通函数实现放 `.cpp`。
- 模板类必须放在头文件中，例如 `RingBuffer`、`StructParser`。

---

### 6.2 include 顺序

推荐顺序：

```cpp
#include "serial.hpp"              // 当前模块头文件

#include <cstdint>                  // C/C++ 标准库
#include <string>
#include <vector>

#include <boost/asio.hpp>           // 第三方库

#include "tools/logger.hpp"         // 项目内部头文件
```

规则：

1. 当前 `.cpp` 对应的 `.hpp` 放第一行。
2. 标准库第二组。
3. 第三方库第三组。
4. 项目内部头文件最后。
5. 不要依赖 include 的“偶然传递”。用到什么就显式 include 什么。

## 8. 日志规范

项目已有 `tools::Logger`，正式代码统一使用日志宏：

```cpp
static constexpr const char* MODULE = "SERIAL";

LOG_DEBUG(MODULE, "rx len={}", len);
LOG_INFO(MODULE, "{} open success", device);
LOG_WARN(MODULE, "crc check failed");
LOG_ERROR(MODULE, "{} open failed: {}", device, e.what());
```

日志等级使用建议：

| 等级        | 使用场景                             |
| ----------- | ------------------------------------ |
| `LOG_DEBUG` | 高频调试信息、变量值、临时排查信息   |
| `LOG_INFO`  | 初始化成功、设备打开成功、状态切换   |
| `LOG_WARN`  | CRC 失败、短暂丢帧、可恢复异常       |
| `LOG_ERROR` | 设备打开失败、配置缺失、不可恢复异常 |

要求：

- 每个 `.cpp` 文件建议定义自己的 `MODULE`。
- `MODULE` 名称大写，例如 `SERIAL`、`CAMERA`、`AUTO_AIM`。
- 正式代码不要直接 `std::cout` 打印运行信息。
- 测试程序可以使用 `std::cout`，但关键错误仍建议用 Logger。
- `LOG_DEBUG` 在 Release 模式下会被编译期移除，不能依赖它产生副作用。

## 9. 错误处理规范

### 9.1 设备打开/初始化

设备类初始化函数推荐返回 `bool`：

```cpp
bool Camera::open(const CameraConfig& config);
bool Serial::open(const std::string& device, int baudrate);
bool FoxGloveComm::reset(const std::string& host, uint16_t port);
```

要求：

- 成功返回 `true`。
- 失败返回 `false`，同时输出 `LOG_ERROR`。
- 不要静默失败。

---

### 9.2 异常处理

不推荐空的 `catch (...)`：

```cpp
catch (...) {
    is_open_ = false;
    return 0;
}
```

推荐至少记录日志：

```cpp
catch (const std::exception& e) {
    is_open_ = false;
    LOG_ERROR(MODULE, "write failed: {}", e.what());
    return 0;
} catch (...) {
    is_open_ = false;
    LOG_ERROR(MODULE, "write failed: unknown exception");
    return 0;
}
```

## 10. 串口协议规范

当前 `io::Serial` 的设计是“结构体驱动串口通信”：

- 数据包用 `struct` 描述。
- 前 2 字节作为帧头。
- 内部使用环形缓冲区防粘包/半包。
- 收到完整结构体后通过回调交给用户。
- CRC 校验由上层回调中完成。

推荐数据包格式：

```cpp
struct __attribute__((packed)) RecvPackage {
    uint8_t head[2] = {'T', 'G'};
    uint32_t data;
    uint16_t crc16;
};
```

要求：

- 串口收发结构体必须显式 `packed`，避免编译器填充字节。
- 结构体第一项必须是 2 字节帧头。
- 多字节数据要明确大小端，尤其是和单片机通信时。
- CRC 字段建议放在结构体最后。
- 回调里先校验 CRC，再使用数据。

示例：

```cpp
serial.recv<RecvPackage>([](const RecvPackage& pkt) {
    if (!tools::check_crc16(reinterpret_cast<const uint8_t*>(&pkt), sizeof(RecvPackage))) {
        LOG_WARN(MODULE, "CRC check failed");
        return;
    }

    LOG_INFO(MODULE, "data={}", pkt.data);
});
```

```cpp
class FoxGloveComm {
public:
    FoxGloveComm(const std::string& host = "0.0.0.0", uint16_t port = 8765);
    ~FoxGloveComm();

    FoxGloveComm(const FoxGloveComm&) = delete;
    FoxGloveComm& operator=(const FoxGloveComm&) = delete;
    FoxGloveComm(FoxGloveComm&&) noexcept;
    FoxGloveComm& operator=(FoxGloveComm&&) noexcept;

    bool ok() const;
    bool reset(const std::string& host, uint16_t port);
};
```

这是比较好的资源管理风格，后续可继续保持：

- 拥有资源的类禁止拷贝。
- 如有必要，允许移动。
- 具体实现放入 `Impl`，减少头文件依赖。
- 失败时使用 Logger 输出错误。

## 14. 注释规范

### 14.1 文件头

当前项目已有类似：

```cpp
//
// Created by Interweave on 2026/4/10.
//
```

建议统一为：

```cpp
/**
 * @file serial.hpp
 * @brief 结构体驱动串口通信模块。
 */
```

### 14.2 函数注释

对 public API 使用 Doxygen 风格：

```cpp
/**
 * @brief 打开串口设备。
 * @param device 设备路径，例如 /dev/ttyACM0。
 * @param baudrate 波特率，例如 2000000。
 * @return true 打开成功；false 打开失败。
 */
bool open(const std::string& device, int baudrate);
```

注释原则：

- 注释解释“为什么这样做”，不是重复代码做了什么。
- 协议、单位、坐标系、线程安全要求必须写清楚。
- 临时代码用 `TODO(name): reason` 标记。

示例：

```cpp
// TODO(Interweave): 将 CRC 校验从回调移动到协议层，避免每个业务模块重复写。
```

## 15. 第三方依赖管理规范

当前项目依赖：

- Boost / Boost.Asio
- OpenCV
- Aravis
- libusb
- toml++
- Foxglove SDK / library

规范：

- 系统依赖写在 `README.md`。
- 单头文件库可以放在 `tools/`，例如 `tomlpp.hpp`。
- 较大的第三方库放在独立目录，例如 `tools/foxglove/`。
- 自己写的封装层和第三方源码分开，例如：
    - 自己写：`tools/foxglove_comm.hpp/.cpp`
    - 第三方：`tools/foxglove/`
- 不要直接在业务代码里到处包含第三方库细节，尽量通过自己的封装类隔离。

## 16. 测试代码规范

测试文件放在：

```text
task/test/
```

命名：

```text
test_logger.cpp
test_serial.cpp
test_camera.cpp
test_foxglove.cpp
read_toml.cpp
```

要求：

- 一个测试文件只测试一个模块。
- 测试文件可以包含临时路径和实验代码，但正式任务代码不应复制测试代码。
- 测试用例能复现问题时，尽量保留最小可运行版本。
- 每个测试目标都应在 `task/CMakeLists.txt` 中加入 `add_executable()`。

## 19. 简短总结

本项目建议长期保持以下结构：

```text
config：参数
io：设备和通信
tools：通用工具
app：算法和业务
task：入口和测试
```

核心原则：

- 驱动层不要写业务。
- 工具层不要依赖业务。
- 参数不要硬编码。
- 正式代码统一用 Logger。
- 第三方库和自写封装分开。
- 新模块必须有清晰的 `.hpp/.cpp/CMakeLists.txt/README.md` 组织方式。
